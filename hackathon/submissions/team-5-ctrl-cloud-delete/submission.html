🏦 The Problem

Contoso Financial runs three critical on-prem workloads on a fragile Flask + SQLite monolith — a customer web app, a nightly batch reconciliation job, and a shared reporting database queried by 5 internal teams. Single point of failure. No scaling. No separation of concerns.

Before
  • 1 Flask process, 1 SQLite file
  • No horizontal scaling
  • Batch job locks the DB
  • 5 teams share one file
  • Zero fault isolation
After
  • 6 containerized services
  • Scale backend to N replicas
  • Async queue-driven worker
  • Postgres concurrent reads
  • Each service fails independently

⚔️ Stakeholder Tension

CFO Priority

Just signed a cloud contract. Needs to exit on-prem fast to realize cost savings. Wants results this quarter, not next year.

CTO Priority

No lift-and-shift. Wants genuine cloud-native architecture — managed services, autoscaling, event-driven patterns — not just a VM in the cloud.

Resolution: Mixed migration strategy — re-platform web app + DB fast (CFO ✓), fully refactor the batch job to event-driven (CTO ✓).

📋 Migration Decisions

Workload Pattern Rationale
Customer Web App Re-platform Containerized, stateless, Nginx-fronted. Fast to ship — CFO ✓. Horizontally scalable — CTO ✓.
Batch Reconciliation Refactor Event-driven worker triggered by Redis queue. Outputs to MinIO (S3). Fully decoupled.
Reporting Database Re-platform SQLite → PostgreSQL. Schema unchanged. Solves 5-team concurrent read contention.
